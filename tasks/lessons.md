# Lessons

## 2026-07-15: Magic Trackpad のペアリング仕様(最終結論)

- Magic Trackpad はペアリング情報を1台分しか保持しない。別の Mac とペアリングすると古い側のリンクキーは無効になり、接続は認証エラーで失敗し続ける
- 待機中(切断中)の Magic Trackpad は、無線でのペアリング要求を受け付けない。IOBluetoothDevicePair はエラー 0x2 で即失敗する。GUI セッションでも SSH セッションでも同じ。inquiry(探索)にも応答しない
- **接続中にペアリング解除(IOBluetoothDevice の remove)すると、解除がトラックパッドに伝わり発見可能モードに入る**。これが唯一の無線切り替え経路。切断してから解除してもトラックパッドには伝わらない
- よって切り替え = 「持ち主が接続中に release → 受け取る側が即 pair + connect(リトライ必須。1回目の pair は失敗しやすい)」
- 教訓1: 「自動でペアリングし直せばよい」という仮説を、実装前に1コマンドの実験で検証すべきだった
- 教訓2: 動作原理が不明な領域では、同じ課題を解いている先行実装のソースを最初に読む。今回も試行錯誤を大幅に減らせたはずだった

## 2026-07-15: TCC(Bluetooth権限)の罠

- 権限がない場合、IOBluetoothDevice.pairedDevices() はエラーではなく空配列を返す。「ペアリングされていない」と誤診断しやすい
- 対策: CBManager.authorization で権限状態を明示的に確認する(cli selfcheck)
- SSH 経由のプロセスの Bluetooth 権限は /usr/libexec/sshd-keygen-wrapper に付与する

## 2026-07-15: 検証時の破壊的操作

- ペアリング削除(unpair)を伴う実験は、失敗するとデバイスが孤立する。先に復旧手順を確保してから実行すること
- pair() の実装も「先に unpair」ではなく「pair 失敗時のみ unpair して一度だけやり直す」順序にした

## 2026-07-23: 不安定な非同期処理には進捗表示をセットで考える

- 「切り替えが不安定」という報告への対策(リトライ強化・接続確認の安定性チェックなど)だけを計画したところ、「どういう状態なのかユーザーにわかるようにしてほしい」と修正が入った
- 数秒〜十数秒かかる非同期処理(特に失敗しうるもの)は、内部ロジックの堅牢化だけでなく「今何をしているか」をUIに出すこと自体が体験改善になる。「切り替え中…」のような単一の busy 表示だけでは、進んでいるのか固まっているのか区別できず不安・苛立ちにつながる
- 対策: `SwitchEngine` に `onProgress` コールバックを持たせ、各ステップ(解除中・n/m回目のペアリング試行・復旧中など)をUI(`AppState.progressText`)にリアルタイム反映する形にした
- 教訓: 「バグを直す」計画を立てるときも、直す対象がユーザーに見える処理なら、修正が効くまでの体感(進捗の可視性)もあわせて設計する
